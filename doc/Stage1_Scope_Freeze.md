# InsightGraph Stage 1 范围冻结文档

## 1. 文档目的

本文件用于冻结 InsightGraph `Stage 1` 的产品范围、技术边界、运行方式、人工确认流程和待办事项。

本文件中的结论来自逐项确认后的最终决定，后续如需变更，必须追加记录，而不是口头覆盖。

## 2. Stage 1 总目标

Stage 1 的目标是构建一个**本地单用户研究日报系统**，支持：

- arXiv 配置化抓取
- `abstract` 初筛
- 入选论文全文精读
- 证据抽取
- 知识链接
- Markdown 日报归档
- 本地 Web 图形化管理台
- 人工审核与修正
- 手动飞书推送

Stage 1 首发重点服务于以下分类：

- `cs.AI`
- `cs.CV`
- `stat.ML`

系统必须设计为**可配置扩展**，后续允许通过配置增加更多学科与分类，但 Stage 1 默认范围按上述分类冻结。

## 3. Stage 1 纳入范围

Stage 1 明确纳入：

- arXiv 抓取
- 配置化 `categories / topics`
- 去重与版本归并
- 基础标签与评分
- 日报 Markdown 生成
- 历史归档与回查
- 飞书单渠道推送
- 入选论文全文解析与精读级判断
- 入选论文知识链接
- 本地单用户图形化管理台
- 人工审核、人工修正、单篇重解析、按日期重跑

## 4. Stage 1 不纳入范围

Stage 1 明确不纳入：

- 周报
- 专题观察
- 突破追踪
- 自由对话式精读
- 多渠道推送
- 多用户
- 登录鉴权
- 复杂权限系统
- 复杂知识图谱可视化
- 富文本编辑器
- 正式公网部署

## 5. 默认产品与输出参数

- 输出语言：中文
- 管理台界面语言：中文
- 每类最大篇数：5
- 输出风格：摘要优先，附简短判断
- 阅读深度：标准
- 高质量论文不足时：允许少于 5 篇，不强行凑数
- 时区：`Asia/Shanghai`
- 每日自动生成时间：`09:00`
- 首轮初始化回填窗口：最近 `3` 天

## 6. 推送策略

- 首发推送渠道：飞书
- 飞书推送方式采用**自建应用机器人私聊**
- 使用：
  - `App ID`
  - `App Secret`
  - `tenant_access_token`
  - 目标用户 `open_id`
- 飞书主要发送摘要提醒
- 完整内容以 Markdown 归档和本地 Web 管理台承载
- 如附带链接，只要求在本机环境中可用，不要求远程可访问
- Stage 1 采用：
  - 先自动生成日报
  - 你在管理台审核 / 修正
  - 再手动触发飞书推送
- 自动任务默认只负责生成，不自动外发

## 7. 模型策略

### 7.1 模型架构要求

- 模型层必须做成多厂商可扩展适配层
- 优先支持国内可用模型
- 同时预留国外模型接口
- 同时支持本地 `Ollama`

### 7.2 Stage 1 优先接入模型

Stage 1 优先实际跑通：

1. `MiniMax`
2. `DeepSeek`
3. `Ollama`

其他模型厂商先预留统一接口与配置结构，不要求 Stage 1 完成联调。

### 7.3 Stage 1 中模型的职责边界

Stage 1 允许使用 LLM 做：

- 一句话结论
- 核心问题提炼
- 方法总结
- 主要结果总结
- 值得关注原因
- 局限 / 风险点提炼
- 证据摘要
- 知识链接理由生成
- 选入理由 / 排序理由生成

Stage 1 不要求模型承担：

- 突破判断
- 自由对话
- 复杂跨域知识图谱推理

## 8. 内容处理策略

### 8.1 候选论文阶段

- 先基于 `metadata + abstract` 做初筛
- 未入选论文不做全文精读

### 8.2 入选论文阶段

对最终入选论文，必须执行：

- PDF 下载
- 全文抽取
- 分段 / chunking
- 重点证据提炼
- 精读级判断生成

### 8.3 全文失败降级策略

若 PDF 下载失败、解析失败或文本质量过低：

- 该论文仍可入选
- 但必须明确标记 `全文解析失败` 或 `低置信度`
- 内容回退到 `metadata + abstract` 级别判断
- 不允许伪造全文证据
- 单篇失败不得中断整份日报
- 后续允许手动重跑该论文全文处理

## 9. 标签、评分与知识链接边界

### 9.1 标签边界

Stage 1 稳定承诺：

- `category`
- `topic`

Stage 1 不承诺稳定支持：

- `method`
- `task`
- `domain`
- `application`

### 9.2 评分边界

Stage 1 保留以下评分维度：

- `relevance`
- `novelty`
- `credibility`
- `impact`
- `readability`
- `freshness`

要求：

- 排序结果必须附带可解释理由
- 首发允许部分评分采用启发式规则

### 9.3 知识链接边界

Stage 1 即启用知识链接，但范围收缩为：

- 只对最终入选论文生成
- 每篇至少生成 `2` 条知识链接
- 首发稳定支持：
  - `related`
  - `prerequisite`
  - `followup`
- 每条链接必须包含：
  - 目标论文
  - 链接类型
  - 链接理由
  - 置信度
- 置信度不足时允许少生成，不能硬凑
- 暂不稳定承诺：
  - `competing`
  - `cross_domain`

知识链接目标来源限定为：

- 当日抓取论文
- 历史已入库论文

Stage 1 不依赖外部额外学术搜索源，也不要求覆盖全量 arXiv 历史库。

## 10. 日报最低输出标准

每篇入选论文至少包含：

- 标题
- 原文链接
- 一句话结论
- 核心问题是什么
- 方法做了什么
- 主要结果是什么
- 为什么值得关注
- 局限或风险点
- `2-4` 条关键证据
- 分类 / 主题
- 选入理由 / 排序理由
- 全文解析失败时的明确标记

每条关键证据至少包含：

- 证据摘要
- 来源位置
- 该证据支撑了什么判断

分类级别至少包含：

- 分类名称
- 今日新增数
- 入选数
- 本类重点论文列表

## 11. 图形化管理台范围

Stage 1 包含本地单用户图形化管理台，技术实现优先：

- `FastAPI + 服务端模板页`

不引入重前端框架。

### 11.1 首发页面范围

- `Dashboard`
- `Digests`
- `Papers`
- `Paper Detail`
- `Config / Review`

### 11.2 管理台最小能力

- 查看历史日报列表
- 查看单篇论文详情
- 按日期 / 分类 / 主题检索论文
- 查看摘要、精读结论、关键证据、知识链接
- 查看全文解析状态
- 查看待人工确认或已手工修正记录

## 12. 人工修正与运维操作

### 12.1 人工修正范围

Stage 1 允许的人工修正包括：

- 标记论文 `入选`
- 标记论文 `忽略`
- 修正 `topic`
- 修正知识链接：
  - 新增
  - 删除
  - 修改类型
  - 修改理由
- 记录全文解析状态备注

Stage 1 不开放：

- 任意编辑整篇日报正文
- 复杂批量操作
- 自定义评分公式 UI
- 任意编辑底层原始抓取记录

### 12.2 人工修正生效规则

- 人工修正作为高优先级覆盖规则
- 重新生成日报时优先使用人工修正结果
- 原始自动结果必须保留，便于回溯和比较

### 12.3 手动操作能力

Stage 1 必须支持：

- 在 `Paper Detail` 页面手动触发单篇重解析
  - 重新下载 PDF
  - 重新解析全文
  - 重新生成精读结论
  - 重新生成证据
  - 重新生成知识链接
- 按日期手动重跑整天日报
- 将“重跑生成”和“补发推送”拆成两个动作
- 对重跑、补发、归档、推送都要考虑幂等控制

## 13. 工程与运行基线

- Python 3.11
- `uv`
- `FastAPI`
- `SQLAlchemy + Alembic`
- `APScheduler`
- `pytest`
- `ruff`
- 开发阶段使用 `Docker Compose` 启动 `PostgreSQL + pgvector`
- Stage 1 按单用户、本地运行 / 本地联调设计

## 14. 验收与样本策略

### 14.1 样本准备方式

- 由 Codex 准备：
  - `20` 篇正样本候选
  - `10` 篇反样本候选
- 覆盖范围：
  - `cs.AI`
  - `cs.CV`
  - `stat.ML`
- 由你标记：
  - 是否应入选
  - 可选：是否高优先级

### 14.2 Stage 1 验收结论形式

Stage 1 的最终验收结论分为三类：

- `通过`
- `通过，但需修复若干非阻塞问题`
- `不通过，需回到某一子阶段修正`

如不通过，需要明确回退点，例如：

- 抓取与标准化
- 评分与筛选
- 日报与证据呈现
- 推送与幂等

## 15. 仍需你亲自完成的事项

以下事项尚需你亲自准备或完成：

### 15.1 外部资源准备

- 准备 `MiniMax` API Key
- 准备 `DeepSeek` API Key
- 在本机安装并运行 `Ollama`
- 确认本机可运行 Docker Compose
- 准备飞书自建应用机器人配置
  - `App ID`
  - `App Secret`
  - 目标用户 `open_id`

### 15.2 网络与访问准备

- 确认本机可以访问本地 Web 管理台
- MVP 阶段不要求远程或局域网访问

### 15.3 后续人工确认职责

你后续仍需承担：

- Gate 1：范围 / 配置确认
- Gate 2：样本抽检确认
- Gate 3：渠道联调与运行确认
- Gate 4：Stage 1 最终验收
